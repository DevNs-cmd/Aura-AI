```txt G:\ALGOFORCE_AI\AURA\Aura-AI\README_DATABASE_SETUP.md ``` ````markdown # Aura AI Database Infrastructure Setup This document explains the database infrastructure added for the Aura AI project. The database layer uses: - PostgreSQL for permanent structured data - Redis for caching, sessions, temporary context, rate limiting, and WebSocket state - ChromaDB for local vector storage and semantic search - Pinecone as a future production vector database option This setup focuses only on the database/cache/vector layer. It does not include final LLM integration, prompt engineering, or complete AI orchestration. --- ## 1. Infrastructure Overview Aura AI uses three main storage systems: | Layer | Technology | Purpose | |---|---|---| | Primary Database | PostgreSQL | Stores users, chats, messages, memories, documents, and journal entries | | Cache Layer | Redis | Stores sessions, chat context, memory cache, rate limits, temporary AI context, and WebSocket state | | Vector Database | ChromaDB | Stores vector embeddings for memory, documents, and journals | | Future Vector DB | Pinecone | Future production cloud vector database option | --- ## 2. Environment Variables Use the following variables in `.env` for local development. ```env POSTGRES_USER=aura_user POSTGRES_PASSWORD=aura_password POSTGRES_DB=aura_ai POSTGRES_HOST=localhost POSTGRES_PORT=5432 REDIS_HOST=localhost REDIS_PORT=6379 REDIS_URL=redis://localhost:6379 CHROMA_HOST=localhost CHROMA_PORT=8001 VECTOR_DB_PROVIDER=chroma PINECONE_API_KEY= PINECONE_INDEX_NAME= ``` Only `.env.example` should be committed to GitHub. Do not commit the real `.env` file. --- ## 3. Docker Services The `docker-compose.yml` file starts the required database services. ### PostgreSQL Container name: ```txt aura_postgres ``` Port: ```txt 5432 ``` Database: ```txt aura_ai ``` --- ### Redis Container name: ```txt aura_redis ``` Port: ```txt 6379 ``` --- ### ChromaDB Container name: ```txt aura_chromadb ``` Port mapping: ```txt 8001:8000 ``` Local ChromaDB URL: ```txt http://localhost:8001 ``` --- ## 4. Start Database Services From the project root, run: ```powershell docker compose up -d ``` Verify containers: ```powershell docker ps ``` Expected containers: ```txt aura_postgres aura_redis aura_chromadb ``` Check logs: ```powershell docker logs aura_postgres docker logs aura_redis docker logs aura_chromadb ``` --- ## 5. PostgreSQL Setup PostgreSQL is used for permanent application data. Schema file: ```txt app/db/init.sql ``` Seed file: ```txt app/db/seed.sql ``` Main tables: - users - chats - messages - memories - documents - journal_entries --- ## 6. PostgreSQL Verification Open PostgreSQL shell: ```powershell docker exec -it aura_postgres psql -U aura_user -d aura_ai ``` List tables: ```sql \dt ``` Expected tables: ```txt users chats messages memories documents journal_entries ``` Exit PostgreSQL: ```sql \q ``` --- ## 7. Run Seed Data Run seed file: ```powershell docker exec -i aura_postgres psql -U aura_user -d aura_ai < app/db/seed.sql ``` Verify seed data: ```powershell docker exec -it aura_postgres psql -U aura_user -d aura_ai ``` Then run: ```sql SELECT * FROM users; SELECT * FROM chats; SELECT * FROM messages; ``` --- ## 8. PostgreSQL Responsibilities PostgreSQL stores permanent records such as: - User profiles - Chat sessions - Chat messages - User memories - Uploaded document metadata - Journal entries - Embedding IDs linked to ChromaDB records Example relationship: ```txt User ↓ Chat ↓ Messages ``` Example vector mapping: ```txt PostgreSQL memories.id ↓ ChromaDB embedding_id = memory_{memory_id} ``` --- ## 9. Redis Setup Redis is used for fast temporary data. Redis responsibilities: - Session storage - Chat context cache - Memory retrieval cache - Rate limiting - Temporary AI context - WebSocket connection state - Frequently accessed data cache --- ## 10. Redis Key Naming Convention | Purpose | Key Pattern | |---|---| | Session | `session:{user_id}:{device_id}` | | WebSocket State | `ws:{user_id}` | | Chat Context | `chat_context:{chat_id}` | | Memory Cache | `memory_cache:{user_id}:{query_hash}` | | Rate Limit | `rate_limit:{user_id}:{route}:{minute}` | | Temporary AI Context | `ai_temp:{chat_id}` | | Frequently Accessed Data | `frequent:{key_name}` | --- ## 11. Redis TTL Plan | Data Type | TTL | |---|---| | Session | 7 days | | WebSocket state | 1 hour | | Chat context | 30 to 60 minutes | | Memory cache | 10 to 30 minutes | | Rate limit | 60 seconds | | Temporary AI context | 10 minutes | | Frequently accessed data | 5 to 30 minutes | --- ## 12. Test Redis Open Redis CLI: ```powershell docker exec -it aura_redis redis-cli ``` Ping Redis: ```bash PING ``` Expected response: ```txt PONG ``` Set a test key: ```bash SET test_key "Aura Redis Working" GET test_key ``` Check TTL example: ```bash SETEX ttl_test 30 "delete after 30 seconds" TTL ttl_test ``` Exit Redis: ```bash exit ``` --- ## 13. Redis Cache Flow ### Chat Context Cache Flow ```txt User opens chat ↓ Backend checks Redis chat_context:{chat_id} ↓ If found, use Redis data ↓ If not found, fetch messages from PostgreSQL ↓ Save result in Redis ``` ### Memory Cache Flow ```txt User asks question ↓ FastAPI checks memory_cache:{user_id}:{query_hash} ↓ If found, use cached memory ↓ If not found, search ChromaDB ↓ Save result in Redis ``` ### Rate Limiting Flow ```txt User sends many requests ↓ Redis increments rate_limit key ↓ If count is above limit, block request ↓ Key expires after 60 seconds ``` ### WebSocket State Flow ```txt User connects to chat ↓ Save socket ID in Redis ↓ User disconnects ↓ Update status offline ``` Example Redis WebSocket key: ```txt Key: ws:user_123 socket_id: abc123 status: online last_seen: 1783350000000 ``` --- ## 14. ChromaDB Setup ChromaDB is used as the local vector database for the MVP. Service file: ```txt app/ai/vectorstore/chroma_service.py ``` ChromaDB stores vector data for: - User memories - Document chunks - Journal entries --- ## 15. ChromaDB Collections The implementation uses three collections: ```txt aura_memories aura_documents aura_journals ``` --- ### aura_memories Stores long-term user memories. Example metadata: ```json { "user_id": "user_uuid", "source_id": "memory_uuid", "type": "preference" } ``` --- ### aura_documents Stores uploaded document chunks. Example metadata: ```json { "user_id": "user_uuid", "document_id": "document_uuid", "chunk_index": 1, "file_name": "notes.pdf" } ``` --- ### aura_journals Stores journal entries. Example metadata: ```json { "user_id": "user_uuid", "journal_id": "journal_uuid", "mood": "stressed" } ``` --- ## 16. ChromaDB Health Check Start FastAPI: ```powershell python -m uvicorn app.main:app --reload --port 8000 ``` Open Swagger docs: ```txt http://127.0.0.1:8000/docs ``` Test: ```txt GET /health/chroma ``` Expected response: ```json { "status": "connected", "collections": [ "aura_memories", "aura_documents", "aura_journals" ] } ``` --- ## 17. ChromaDB Vector Testing ### Add Memory Vector Endpoint: ```txt POST /vectors/memory/add ``` Example body: ```json { "user_id": "user_123", "memory_id": "memory_001", "content": "User prefers short and clear answers.", "memory_type": "preference" } ``` Expected response: ```json { "success": true, "embedding_id": "memory_memory_001" } ``` --- ### Search Memory Vector Endpoint: ```txt POST /vectors/memory/search ``` Example body: ```json { "user_id": "user_123", "query": "How does the user like answers?", "limit": 5 } ``` Expected result should return the related memory. --- ### Add Document Vector Endpoint: ```txt POST /vectors/document/add ``` Example body: ```json { "user_id": "user_123", "document_id": "doc_001", "chunk_text": "Aura AI uses PostgreSQL, Redis, and ChromaDB.", "chunk_index": 1, "file_name": "architecture_notes.pdf" } ``` --- ### Search Document Vector Endpoint: ```txt POST /vectors/document/search ``` Example body: ```json { "user_id": "user_123", "query": "Which database is used for caching?", "limit": 5 } ``` --- ### Add Journal Vector Endpoint: ```txt POST /vectors/journal/add ``` Example body: ```json { "user_id": "user_123", "journal_id": "journal_001", "content": "Today I felt stressed because of project deadlines.", "mood": "stressed" } ``` --- ### Search Journal Vector Endpoint: ```txt POST /vectors/journal/search ``` Example body: ```json { "user_id": "user_123", "query": "When did the user feel stressed?", "limit": 5 } ``` --- ## 18. PostgreSQL and ChromaDB ID Mapping PostgreSQL stores permanent metadata. ChromaDB stores semantic vector data. Example: ```txt PostgreSQL memory record: memories.id = 123 ChromaDB vector record: embedding_id = memory_123 ``` The generated `embedding_id` can be saved back into PostgreSQL. Example: ```sql UPDATE memories SET embedding_id = 'memory_123' WHERE id = '123'; ``` --- ## 19. Pinecone Future Option Pinecone is not implemented in the MVP.
