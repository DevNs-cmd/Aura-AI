## 2. Architecture Overview

### 2.1 Single Source of Truth

```
lib/core/localization/supported_languages.dart
```

Defines `const List<SupportedLanguage> supportedLanguages` â€” the registry. Every consumer
reads from this list at runtime; there are no parallel language lists in the codebase.

### 2.2 Locale Controller

```
lib/core/localization/locale_controller.dart
```

- `LocaleNotifier` extends `StateNotifier<Locale>`.
- `setLocale()` rejects any locale not present in `SupportedLanguagesRegistry` (gated).
- Persists locale code to `SharedPreferences` using key `'language'`.
- Restores locale on boot via `_loadPersistedLocale()`.

### 2.3 ARB-Based Translation Files

```
lib/l10n/app_en.arb   (template â€” 401 keys)
lib/l10n/app_es.arb   (Spanish)
lib/l10n/app_hi.arb   (Hindi)
```

Config (`l10n.yaml`):
```yaml
arb-dir: lib/l10n
template-arb-file: app_en.arb
output-class: AppLocalizations
output-dir: lib/core/localization/generated
```

### 2.4 MaterialApp Wiring (`main.dart`)

```dart
locale: appLocale,                                   // driven by localeProvider
supportedLocales: AppLocalizations.supportedLocales, // auto-generated by flutter gen-l10n
localizationsDelegates: [AppLocalizations.delegate, ...]
```

`AppLocalizations.supportedLocales` is auto-derived from ARB files by `flutter gen-l10n`.
It is **not** manually maintained.

### 2.5 Language Selectors (Settings + Mood Screen)

Both selectors iterate `supportedLanguages` dynamically:
```dart
// settings_screen.dart line 385
children: supportedLanguages.map((lang) { ... }).toList()

// mood_selection_screen.dart line 449
...supportedLanguages.map((lang) { ... })
```
No UI file contains a hardcoded list of language options.

### 2.6 Validation Tooling

`tool/arb_validator.dart` discovers ARB files dynamically via filesystem regex:
```dart
final arbRegex = RegExp(r'^app_([a-zA-Z_]+)\.arb$');
```
No hardcoded locale list â€” picks up any `app_XX.arb` file automatically.

`test/localization_test.dart` iterates `supportedLanguages` dynamically in all 5 test cases.

## 7. Final Verdict

```
PASS â€” LANGUAGE ADDITION IS TRANSLATION-FILE + REGISTRY-ENTRY ONLY
```

### Required Steps to Add a New Production Language:
1. Create `lib/l10n/app_XX.arb` (where `XX` is the BCP-47 language tag, e.g., `de`, `pt`, `ar`)
2. Add one `SupportedLanguage` entry to `lib/core/localization/supported_languages.dart`:
   ```dart
   SupportedLanguage(
     locale: Locale('XX'),
     nativeName: 'NativeName',
     englishName: 'EnglishName',
   ),
   ```
3. Run `flutter gen-l10n`

**No other production source files require modification.**

### Notes for Future Phases:
- Phase 2 (Responsive UI) should address label wrapping for longer translations.
- The 6 Gaps listed in Section 6 can be addressed as part of a general localization hardening pass.
- The `_pendingLanguage = 'en'` default can be improved to read from `localeProvider.state.languageCode`.
- When removing a language, `app_localizations_XX.dart` must be manually deleted after removing
  the ARB file (`flutter gen-l10n` does not auto-remove orphaned locale files).
